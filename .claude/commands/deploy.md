# デプロイ — コミット・プッシュ・本番確認

コミット、GitHub Pushし、本番サイトの稼働を確認するスキル。

## 引数

`$ARGUMENTS` — 省略可。指定時はコミットメッセージとして使用（conventional prefix付き）。

## 手順

### 1. 変更状況の確認

- `git status` で現在の変更を確認する
- `git diff --stat` で変更ファイルのサマリーを表示する
- 変更がなければ「コミットする変更はありません」と報告して終了する

### 2. ファイルのステージング

- プロジェクトに関係するファイルのみステージングする
- 対象: `*.html`, `css/`, `js/`, `images/`, `.claude/`, その他プロジェクト関連ファイル
- **除外**: `mysql*.sql`, `static/`, `www/`, `node_modules/`, `.env` など非プロジェクトファイル
- `git add` で個別にファイルを追加する（`git add .` は使わない）

### 3. コミットメッセージの作成

- `$ARGUMENTS` が指定されている場合: それをコミットメッセージとして使用する
- `$ARGUMENTS` が空の場合: 変更内容を分析して適切なメッセージを作成する
- メッセージは英語で、conventional prefix を付ける:
  - `feat:` — 新機能
  - `fix:` — バグ修正
  - `style:` — 見た目の変更（CSSなど）
  - `chore:` — 設定やメンテナンス
  - `refactor:` — リファクタリング
  - `docs:` — ドキュメント

### 4. コミットの実行

- `git commit` でコミットする
- Co-Authored-By ヘッダーを付与する

### 5. プッシュ

- `git push origin master` でリモートにプッシュする
- プッシュが失敗した場合はエラーを報告し、対処法を提案する

### 6. 本番サイトの確認

- 数秒待ってから本番URLにアクセスして応答を確認する
- 本番URL: `https://yuyasuzuki117.github.io/ninnin-recruit-demo/`
- HTTPステータスコードを確認する（200 OK が期待値）

### 7. 報告

以下の情報をまとめて報告する:

- コミットハッシュ（短縮形）
- 変更ファイル数と一覧
- コミットメッセージ
- プッシュ結果
- 本番サイトの応答状況
- 本番URL
